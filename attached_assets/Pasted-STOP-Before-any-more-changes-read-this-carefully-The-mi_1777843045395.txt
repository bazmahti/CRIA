## The misguided fix

You're about to add a `[services.production]` block to
`artifacts/api-server/.replit-artifact/artifact.toml` to "deploy" the
Node API server. **This will not work** and please do not do it.

The repo's `.replit` sets `deploymentTarget = "autoscale"`. Autoscale
deployments expose **exactly one port** to the public. Adding a
production config to the api-server artifact won't change that —
autoscale will still only run one service. The api-server will either
not start, or it will start and not be reachable.

This is already documented in the codebase. See the comment in
`artifacts/cria-unified/main.py` above the `/api/research/unified`
routes:

> "In production, autoscale only allows ONE exposed port. The
> api-server (Node/Express) is therefore not deployed; these routes
> replace the two endpoints the React dashboard calls so everything
> works through this single FastAPI process."

A previous session correctly diagnosed this and added shims for the
two unified-research endpoints inside `main.py`. The bug is that the
React dashboard calls **more endpoints than just those two**, and the
rest were never shimmed — so every page except `/unified` shows blank
in production.

### Step 2 — Add the missing /api/* shims to main.py

The proper fix is: the deployed FastAPI process (`cria-unified/main.py`)
becomes the single backend, and it shims **every** route the React
dashboard hits. Read these files first to enumerate the full surface:

- `lib/api-spec/openapi.yaml` — canonical route list
- `artifacts/api-server/src/routes/experiments.ts` — handler logic to port
- `artifacts/api-server/src/routes/parallel.ts` — handler logic for /api/research/parallel
- `artifacts/api-server/src/db/schema.ts` (or wherever Drizzle's tables live) — schema for `experiments` and `findings` tables
- `artifacts/cria-unified/main.py` — where the new shims will live, alongside the existing `/api/research/unified` ones

The endpoints to shim, in priority order (highest = visit-first pages):

1. `GET  /api/experiments/summary`             — Control Room landing
2. `GET  /api/experiments`                     — Experiment Queue
3. `GET  /api/experiments/{id}`                — Experiment Detail
4. `GET  /api/experiments/{id}/findings`       — Experiment Detail
5. `GET  /api/experiments/cross-view`          — Findings Index
6. `GET  /api/experiments/reflexivity-report`  — Reflexivity Report
7. `POST /api/experiments`                     — New Experiment
8. `POST /api/experiments/{id}/run`            — Run button
9. `POST /api/experiments/{id}/validate`       — YAML validate
10. `PATCH /api/experiments/{id}`              — Approve/edit
11. `DELETE /api/experiments/{id}`             — Delete
12. `GET  /api/research/parallel`              — Parallel page list
13. `GET  /api/research/parallel/{jobId}`      — Parallel page detail
14. `POST /api/research/parallel`              — Parallel page submit (calls v2 + v4 — keep this as a NotImplemented 501 if those services aren't deployed; do NOT silently fail)
15. `GET  /api/healthz`                        — frontend probe

Implementation rules:

- Use **psycopg** (v3, sync) called via the existing `_db_executor`
  pool, matching the pattern already used by `_db_create_job` /
  `_db_get_job`. Do **not** introduce SQLAlchemy or asyncpg — keep
  the dependency surface small.
- Read from the **same Postgres database** the Drizzle ORM uses
  (same `DATABASE_URL` env var). The `experiments` and `findings`
  tables already exist there — do not recreate them, do not run
  migrations from Python.
- Match the response shape of the Express handlers exactly. The
  React client is generated from the OpenAPI spec via Orval; if a
  field name or casing differs, the dashboard will silently break.
  Test response shape against `lib/api-client-react/src/generated/api.ts`
  type definitions.
- For YAML parsing/validation in `POST /api/experiments`, use
  `pyyaml` and port the validation logic from
  `artifacts/api-server/src/lib/validate-artefact.ts`. Do not skip
  validation — the dashboard surfaces the errors.
- For the run-experiment simulated flow (3-second auto-complete with
  synthetic findings), port `generateFindingsMarkdown` from
  `experiments.ts` to Python. Use FastAPI `BackgroundTasks`, same as
  the existing unified-research job pattern.
- Group the new code in a clearly delimited section in `main.py`,
  same style as the existing `/api/research/unified` shim block:

  ```python
  # ---------------------------------------------------------------------------
  # /api/experiments — production shim (single-port autoscale mode)
  # Mirrors artifacts/api-server/src/routes/experiments.ts
  # ---------------------------------------------------------------------------
  ```
